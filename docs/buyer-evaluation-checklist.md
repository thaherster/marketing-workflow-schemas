# Buyer Evaluation Checklist

## Purpose

This checklist is for teams evaluating whether Draftroom.works workflow patterns can replace fragmented creative review operations.

## Workflow Fit

- [ ] We can identify one high-volume asset family to standardize first.
- [ ] We know which statuses matter operationally outside the creative team.
- [ ] We can name the required reviewer roles for internal and external approvals.
- [ ] We want proof-based review to happen in a single system of record.

## Approval Fit

- [ ] We need to distinguish feedback from true approval.
- [ ] We need explicit approval ownership by role.
- [ ] We need clear version lineage from `v1` to final sign-off.
- [ ] We need a documented policy for late or silent approvers.

## Integration Fit

- [ ] We need Draftroom.works events to update PM or campaign systems.
- [ ] We need proof URLs and version labels retained downstream.
- [ ] We need webhook-friendly event payloads for engineering.
- [ ] We need approval events to be auditable.

## Access and Scale Fit

- [ ] We need a repeatable reviewer access model that can scale across clients or teams.
- [ ] Flat Seats planning matters for how we invite external reviewers.
- [ ] We expect multiple stakeholders to review the same asset at different stages.

## Operational Readiness

- [ ] We can enforce a canonical Asset Status model.
- [ ] We can stop treating chat or email threads as the approval source of truth.
- [ ] We have an owner for workflow governance.
- [ ] We have an owner for integration reliability.

## Signs This Repo Is Relevant To You

This repository will likely be useful if:

- your team is already creating work in design or editing tools but lacks structured review governance
- you need machine-readable approval state, not just creative markup
- you want implementation examples before committing engineering time
